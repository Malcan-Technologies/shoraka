SECTION 3 - Big picture flow diagram
================================================================================

Application
-> Admin review
-> Offer accepted/signed
-> Contract/invoice approved
-> Note created
-> Note published
-> Investor funding
-> Close funding
-> Issuer disbursement withdrawal created
-> Shoraka STP certificate required
-> Trustee letter generated
-> Trustee letter submitted
-> Mark disbursed
-> Note active
-> Repayment received
-> Settlement preview
-> Settlement approve
-> Settlement post
-> Investor balances credited
-> Issuer residual returned if any

Simple money story:

1. Investors put money into the note.
2. At funding close, Cashsouk deducts platform and facility fees.
3. The issuer gets only the net amount.
4. Shoraka is submitted for the gross funded amount.
5. Trustee letter pays the net issuer amount.
6. Issuer/paymaster repays into the repayment pool.
7. Settlement splits repayment to investors, platform service fee, late-fee accounts, and issuer residual.

14A.1 - Background job bootstrap
--------------------------------------------------------------------------------

Script/job name:
- Background job bootstrap.

File path:
- apps/api/src/lib/jobs/index.ts
- apps/api/src/index.ts starts it with initJobs().

How it is triggered:
- API server startup.
- When the API starts, initJobs() registers cron schedules.

Business flow affected:
- Offer expiry.
- Note listing expiry.
- CTOS KYB retry.
- Notification cleanup.

DB tables read/written:
- Depends on each child job.

- Depends on child job.

14A.2 - Note listing / funding expiry job
--------------------------------------------------------------------------------

Script/job name:
- Note listing expiry job.

File paths:
- apps/api/src/lib/jobs/note-listing-expiry.ts
- apps/api/scripts/run-note-listing-expiry.ts

How it is triggered:
- Hourly cron from apps/api/src/lib/jobs/index.ts:
  cron schedule = 0 * * * *
- Manual command from apps/api/package.json:
  pnpm --filter @cashsouk/api run-note-listing-expiry

Purpose:
- Finds open notes where:
  1. note_listings.closes_at has passed, OR
  2. note is fully funded.
- If funded amount meets minimum funding threshold, it calls noteService.closeFunding().
- If funded amount is below minimum funding threshold, it calls noteService.failFunding().

Expiry field:
- note_listings.closes_at controls listing expiry.
- notes.funding_closed_at records when funding was closed.
- notes.funding_closed_at is not the expiry trigger.

Tables read:
- notes
- note_listings

Tables written if note is funded:
- notes
- note_investments
- withdrawal_instructions
- contracts if facility fee is charged
- note_ledger_entries
- note_events
- note_admin_actions
- notifications through notification service

Tables written if note fails:
- notes
- note_investments
- investor_balances
- investor_balance_transactions
- note_events
- note_admin_actions
- notifications through notification service

Statuses changed if funded:
- notes.status: PUBLISHED -> FUNDING
- notes.funding_status: OPEN -> FUNDED
- notes.listing_status: PUBLISHED -> CLOSED
- note_investments.status: COMMITTED -> CONFIRMED
- withdrawal_instructions.status: DRAFT for the new issuer disbursement withdrawal

Statuses changed if failed:
- notes.status: PUBLISHED -> FAILED_FUNDING
- notes.funding_status: OPEN -> FAILED
- notes.listing_status: PUBLISHED -> CLOSED
- note_investments.status: COMMITTED -> RELEASED

Money/bucket movement if funded:
- Same as close funding.
- INVESTOR_POOL DEBIT funded amount.
- OPERATING_ACCOUNT CREDIT platform fee.
- OPERATING_ACCOUNT CREDIT facility fee if contract financing and fee is due.
- ISSUER_PAYABLE CREDIT net issuer disbursement.
- Creates ISSUER_DISBURSEMENT withdrawal.

Money/bucket movement if failed:
- No note ledger bucket movement.
- InvestorBalance is credited back for each released commitment.
- investor_balance_transactions direction = IN, source = NOTE_INVESTMENT_RELEASE.

Shoraka behavior:
- The expiry job does not create shoraka_trade_orders.
- Shoraka row is created only when admin clicks Submit Shoraka Order.

Idempotency:
- The job only selects notes where status = PUBLISHED and funding_status = OPEN.
- After the first successful run, the note is no longer OPEN.
- Running again should not close/fail the same note again.
- closeFunding checks existing ISSUER_DISBURSEMENT withdrawal before creating one.
- failFunding refund transactions use idempotency keys:
  investor-balance:release:fail-funding:<investmentId>

Manual UAT case A - expired listing with zero investment:

Setup data:
1. Create and publish a note with target RM100,000.
2. Do not invest.
3. Set note_listings.closes_at to a past timestamp.

Run trigger:
1. Run:
   pnpm --filter @cashsouk/api run-note-listing-expiry

Expected UI result:

SECTION 15 - Bucket reconciliation checklist
================================================================================

InvestorBalance

Before investment:
- Investor A available balance = starting A.
- Investor B available balance = starting B.

After investment:
- Investor A available = starting A - 60000.
- Investor B available = starting B - 40000.
- Transactions:
  A OUT 60000 NOTE_INVESTMENT_COMMIT.
  B OUT 40000 NOTE_INVESTMENT_COMMIT.

After failed funding:
- Investor balance increases by full committed amount.
- Direction = IN.
- Source = NOTE_INVESTMENT_RELEASE.

After settlement:
- Investor A available increases by principal + net profit + investor ta'widh share.
- Investor B available increases by principal + net profit + investor ta'widh share.

INVESTOR_POOL

After funding close:
- DEBIT funded amount RM100,000.
- Meaning: investor pool cash is moved into fees and issuer payable.

After settlement:
- CREDIT investor principal.
- CREDIT investor net profit.
- CREDIT investor ta'widh share if any.

OPERATING_ACCOUNT

After funding close:
- CREDIT platform fee RM3,000.
- CREDIT facility fee RM1,000.

After settlement:
- CREDIT service fee RM1,500 if 365-day example.

ISSUER_PAYABLE

After close funding:
- CREDIT net issuer amount RM96,000.

After mark disbursed:
- DEBIT RM96,000.
- Net disbursement payable should be zero.

After settlement with residual:
- CREDIT residual amount.

After residual complete:
- DEBIT residual amount.
- Net residual payable should be zero.

REPAYMENT_POOL

When repayment received:
- CREDIT repayment receipt RM110,000.

When settlement posted:
- DEBIT investor principal.
- DEBIT investor net profit.
- DEBIT service fee.
- DEBIT tawidh.
- DEBIT gharamah.
- DEBIT issuer residual.

Manual check:
- REPAYMENT_POOL debits from settlement should equal receipt after any receipt adjustment.

TAWIDH_ACCOUNT

Only applies if ta'widh late compensation is included.

Expected:
- REPAYMENT_POOL DEBIT total ta'widh.
- INVESTOR_POOL CREDIT investor share of ta'widh.
- TAWIDH_ACCOUNT CREDIT retained ta'widh account share.

GHARAMAH_ACCOUNT

Only applies if gharamah penalty is included.

Expected:
- REPAYMENT_POOL DEBIT gharamah.
- GHARAMAH_ACCOUNT CREDIT gharamah.

Global bucket checks:

1. Funding close:
   funded amount = platform fee + facility fee + net issuer disbursement.

2. Settlement:
   receipt = investor principal + investor net profit + service fee + tawidh + gharamah + issuer residual.

3. Investor split:
   sum investor credits = investor principal + investor net profit + investor ta'widh share.

STOP AND INVESTIGATE if any sum does not balance to the cent.
