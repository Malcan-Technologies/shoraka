================================================================================
AFTER OFFER TO NOTE MONEY FLOW — FEATURE-BY-FEATURE FRONTEND MANUAL UAT GUIDE
================================================================================

Purpose:
This guide is for slow manual frontend testing. It is organized by feature, not by
separate theory sections. For each feature, check the screen, amount, calculation,
bucket movement, visible activity/history, and stop rules together.

Important testing rule:
Do not move to the next feature until the STOP POINT for the current feature is
complete.

Fixed UAT numbers used throughout:
- Target funded amount: RM100,000
- Investor A investment: RM60,000
- Investor B investment: RM40,000
- Profit rate: 10%
- Platform fee: 3%
- Facility fee: 1% if contract financing applies
- Service fee: 15% of gross investor profit
- Normal repayment example: RM110,000
- Residual repayment example: RM112,000
- Partial funding example: RM80,000
- Failed funding example: RM50,000

Simple money memory:
- Gross funded amount is the full investor-funded amount.
- Net issuer payout is what issuer receives after platform/facility fees.
- Shoraka uses gross funded amount.
- Trustee issuer payout uses net issuer payout.
- Platform fee and facility fee happen at funding close.
- Service fee happens later during settlement.
- Ta'widh and Gharamah happen only in late/arrears/default scenarios.








################################################################################
FEATURE 1 — Admin sends offer
################################################################################

What this feature means:
- Admin reviews the issuer application and sends a commercial offer.
- Contract offer means facility-level approval.
- Invoice offer means invoice/note-level financing approval.

Why this feature exists:
- Offer is the business agreement before note creation.
- Admin should not create a note until issuer accepts or signs the offer.

Who is involved:
[ ] Admin
[ ] Issuer
[ ] Investor

Where to check:
Admin:
[ ] Admin portal → Applications → Application list
[ ] Admin portal → Application detail
[ ] Review tabs: Financial, Company, Business & Guarantor, Documents, Contract, Invoice
[ ] Contract tab if contract financing
[ ] Invoice tab if invoice financing

Issuer:
[ ] No required action yet until offer is sent

Investor:
[ ] No investor screen yet

What should be shown before action:
[ ] Status: Submitted, Under review, Contract pending, or Invoice pending depending on workflow
[ ] Amount: approved facility or invoice amount if visible
[ ] Button: Send Contract Offer or Send Invoice Offer only after required tabs are approved/unlocked
[ ] Warning/helper text: locked tabs or missing review requirements if not ready
[ ] Card/section: review tabs and offer section

What to click:
[ ] Admin opens the application
[ ] Admin checks required review tabs
[ ] Admin sends contract offer if contract financing
[ ] Admin sends invoice offer if invoice financing

What should be shown after action:
[ ] New status: Contract Sent, Invoice Offer Sent, Offer Sent, or Waiting for issuer
[ ] New button: offer action should no longer look like a fresh unsent offer
[ ] New card: offer summary if supported
[ ] New amount: offered facility RM1,000,000 or offered invoice amount RM100,000 in this UAT
[ ] Success toast/message: offer sent
[ ] Timeline/activity event if visible: offer sent event

Manual calculation:
Formula:
- No money movement calculation yet.
Example:
- Offered invoice amount should be RM100,000.
- Contract facility should be RM1,000,000 if contract financing is used.
Expected:
- The offer terms match the approved terms.

Bucket movement:
- No bucket movement yet.
- No investor money has been committed.
- No issuer payout exists.

Ledger/activity/frontend history check:
[ ] Finance → Buckets activity if visible: no money movement
[ ] Note timeline if visible: no note yet
[ ] Wallet/transaction history if visible: no investor wallet movement
[ ] Finance queue if visible: no issuer payout, repayment, or service fee queue item
[ ] Application activity/timeline/notification if visible: offer sent

What should NOT happen:
[ ] Wrong button: Create Note should not be usable before issuer accepts/signs
[ ] Wrong amount: offer amount should not exceed approved product/business limits
[ ] Wrong status: application should not jump straight to completed before issuer action
[ ] Wrong bucket movement: no Operating Account, Investor Pool, or Issuer Payable movement
[ ] Wrong ledger/activity: no money ledger entry

STOP AND INVESTIGATE if:
[ ] Offer button appears while required tabs are locked or unapproved.
[ ] Admin can create note before issuer accepts/signs.
[ ] Investor can see anything in marketplace at this stage.
[ ] Any money bucket changes.

FEATURE 6 — Funding close
################################################################################

What this feature means:
- Funding close ends investor commitments and creates the issuer payout preparation.
- This is the first big money split.

Why this feature exists:
- Cashsouk must calculate fees and net issuer payout before trustee payout starts.
- The note is Funded, but not Active yet.

Who is involved:
[ ] Admin
[ ] Issuer
[ ] Investor

Where to check:
Admin:
[ ] Admin portal → Note detail
[ ] Lifecycle/funding progress
[ ] Issuer Disbursement card
[ ] Finance → Buckets
[ ] Finance → Issuer Payouts

Issuer:
[ ] Issuer portal → Notes/Financing funding status if visible

Investor:
[ ] Investor portfolio/investments status if visible

What should be shown before action:
[ ] Status: Published/Open Funding with 100% funded, or ready to close at/above minimum
[ ] Amount: funded amount RM100,000
[ ] Button: Close Funding, or auto-close result if fully funded
[ ] Warning/helper text: below-minimum funding should not close successfully
[ ] Card/section: funding progress

What to click:
[ ] Admin clicks Close Funding if not auto-closed
[ ] Or refresh/check after auto-close

What should be shown after action:
[ ] New status: Funded
[ ] New button: issuer payout/trustee workflow buttons begin
[ ] New card: Issuer Disbursement card
[ ] New amount: platform fee RM3,000, facility fee RM1,000, net issuer payout RM96,000
[ ] Success toast/message: funding closed
[ ] Timeline/activity event if visible: funding closed/funded event

Manual calculation:
Formula:
- Net issuer payout = funded amount - platform fee - facility fee.
Example:
- Platform fee = 100,000 x 3% = RM3,000.
- Facility fee = 100,000 x 1% = RM1,000.
- Net issuer payout = 100,000 - 3,000 - 1,000 = RM96,000.
Expected:
- 100,000 = 3,000 + 1,000 + 96,000.
- Shoraka amount = RM100,000 gross funded.
- Trustee payout amount = RM96,000 net issuer payout.

Bucket movement:
- Investor Pool is used for RM100,000.
- Operating Account gets RM3,000 platform fee.
- Operating Account gets RM1,000 facility fee.
- Issuer Payable gets RM96,000.
- Service fee should NOT appear yet.
- Ta'widh/Gharamah should NOT appear in happy path.

Ledger/activity/frontend history check:
[ ] Finance → Buckets activity if visible: Investor Pool, Operating Account, Issuer Payable
[ ] Note timeline if visible: funding closed
[ ] Wallet/transaction history if visible: investment confirmed/held
[ ] Finance queue if visible: Finance → Issuer Payouts shows issuer disbursement

What should NOT happen:
[ ] Wrong button: settlement/post actions should not be final yet
[ ] Wrong amount: Shoraka should not use RM96,000
[ ] Wrong status: note should not become Active yet
[ ] Wrong bucket movement: service fee, Ta'widh, Gharamah should not move
[ ] Wrong ledger/activity: no repayment or investor return yet

STOP AND INVESTIGATE if:
[ ] Service fee is deducted from issuer payout.
[ ] Platform/facility fee reduces Shoraka amount.
[ ] Issuer payout card appears before funding close.
[ ] Note becomes Active immediately after funding close.
[ ] Fee split does not balance to RM100,000.

FEATURE 8 — Shoraka STP
################################################################################

What this feature means:
- Shoraka STP is proof that the Islamic financing went through commodity trade.
- It is not the payout.

Why this feature exists:
- Issuer disbursement requires Shariah commodity trade proof before Mark Disbursed.
- Shoraka uses gross funded amount.

Who is involved:
[ ] Admin
[ ] Issuer
[ ] Investor

Where to check:
Admin:
[ ] Admin portal → Note detail → Issuer Disbursement card → Shoraka STP box
[ ] Note timeline/activity if visible

Issuer:
[ ] Issuer generally does not operate Shoraka

Investor:
[ ] Investor generally does not see Shoraka controls

What should be shown before action:
[ ] Status: No Shoraka order, or current Shoraka status
[ ] Amount: RM100,000 order_amount / murabaha_amount if shown
[ ] Button: Submit Shoraka Order outside cutoff
[ ] Warning/helper text: cutoff warning between 11:30 PM and 12:30 AM MYT
[ ] Card/section: Shoraka STP box inside Issuer Disbursement card

What to click:
[ ] Submit Shoraka Order
[ ] Query Status
[ ] Fetch Certificate when Completed
[ ] View Certificate after fetch

What should be shown after action:
[ ] New status: Active, Pending Sell, Completed, Cancelled, Take Delivery, Unknown, or certificate fetched state
[ ] New button: Query Status, Fetch Certificate, View Certificate depending on status
[ ] New card: certificate link after fetch
[ ] New amount: Shoraka amount stays RM100,000
[ ] Success toast/message: order submitted/status queried/certificate fetched
[ ] Timeline/activity event if visible: Shoraka order/status/certificate event

Manual calculation:
Formula:
- Shoraka amount = gross funded amount.
Example:
- Gross funded amount = RM100,000.
- Net issuer payout = RM96,000.
Expected:
- Shoraka order amount and murabaha amount use RM100,000, not RM96,000.

Bucket movement:
- No bucket movement from Shoraka.
- Shoraka is proof/audit only.
- No fee deduction happens here.

Ledger/activity/frontend history check:
[ ] Finance → Buckets activity if visible: no movement caused by Shoraka
[ ] Note timeline if visible: Shoraka status/certificate event if supported
[ ] Wallet/transaction history if visible: no investor movement
[ ] Finance queue if visible: issuer payout remains pending until Mark Disbursed

Status meaning:
- Active: order received, trade not complete; query again or wait callback.
- Pending Sell: sell leg pending; query again or contact operations if stuck.
- Completed: trade complete; Fetch Certificate should appear.
- Cancelled: manual review; no disbursement.
- Take Delivery: special/manual state; no automatic disbursement.
- Unknown: unsafe/manual review.
- Callback Completed: may update status, but must not auto-disburse.

What should NOT happen:
[ ] Wrong button: Fetch Certificate should not appear before Completed
[ ] Wrong amount: Shoraka should not use RM96,000
[ ] Wrong status: callback should not mark issuer disbursed
[ ] Wrong bucket movement: no bucket changes from submit/query/fetch
[ ] Wrong ledger/activity: no payout completed event from Shoraka alone

STOP AND INVESTIGATE if:
[ ] Submit is enabled during 11:30 PM–12:30 AM MYT cutoff.
[ ] Shoraka uses net amount RM96,000.
[ ] Mark Disbursed is enabled before certificate exists.
[ ] Callback auto-marks issuer disbursed.
[ ] Cancelled/Take Delivery/Unknown allows payout without manual review.

FEATURE 19 — Late charges / Ta'widh / Gharamah
################################################################################

What this feature means:
- Late charges apply only for late/arrears/default scenarios.
- They are not part of the normal happy path.

Why this feature exists:
- Late repayment may require compensation or penalty/charity allocation.
- Settlement must still balance to the repayment received.

Who is involved:
[ ] Admin
[ ] Issuer
[ ] Investor

Where to check:
Admin:
[ ] Note detail → late/arrears/default area
[ ] Settlement preview/waterfall
[ ] Finance → Buckets
[ ] Arrears/default documents if visible

Issuer:
[ ] Issuer note repayment/late state if visible

Investor:
[ ] Investor return if Ta'widh investor portion is configured

What should be shown before action:
[ ] Status: Late/Arrears/Default-related scenario
[ ] Amount: Ta'widh and Gharamah only if late charge flow is used
[ ] Button: late charge/apply/review/generate arrears/default letter if supported
[ ] Warning/helper text: late/default indicator
[ ] Card/section: late charge panel and settlement waterfall

What to click:
[ ] Open late charge panel if supported
[ ] Preview settlement with late charges
[ ] Generate arrears/default documents if supported

What should be shown after action:
[ ] New status: late charge reflected in settlement preview/post if applied
[ ] New button: next supported late/default action
[ ] New card: Ta'widh/Gharamah rows, arrears/default documents if supported
[ ] New amount: residual reduced by late charges
[ ] Success toast/message: late charge/document action if applicable
[ ] Timeline/activity event if visible: late charge/document event

Manual calculation:
Formula:
- Residual = repayment - principal - net investor profit - service fee - Ta'widh - Gharamah.
Example:
- Repayment RM112,000.
- Principal RM100,000.
- Net investor profit RM8,500.
- Service fee RM1,500.
- Ta'widh RM500.
- Gharamah RM300.
- Residual = 112,000 - 100,000 - 8,500 - 1,500 - 500 - 300 = RM1,200.
Expected:
- 100,000 + 8,500 + 1,500 + 500 + 300 + 1,200 = 112,000.

Bucket movement:
- Ta'widh may partly go to investors and/or Ta'widh Account depending on configuration.
- Gharamah goes to Gharamah Account.
- Both reduce issuer residual.
- Gharamah is not investor profit.

Ledger/activity/frontend history check:
[ ] Finance → Buckets activity if visible: Ta'widh Account and Gharamah Account rows
[ ] Note timeline if visible: late/arrears/default activity
[ ] Wallet/transaction history if visible: Ta'widh investor portion only if configured
[ ] Finance queue if visible: arrears/default documents if supported

What should NOT happen:
[ ] Wrong button: happy path should not show late-charge action as required
[ ] Wrong amount: late charges should not make total split exceed repayment
[ ] Wrong status: normal on-time note should not become arrears/default
[ ] Wrong bucket movement: Gharamah should not go to investor as normal profit
[ ] Wrong ledger/activity: late charges should not appear in normal happy path

STOP AND INVESTIGATE if:
[ ] Ta'widh/Gharamah appear in normal non-late happy path.
[ ] Gharamah is paid as investor profit.
[ ] Total split exceeds repayment received.
[ ] Investor principal changes because of late charges.
