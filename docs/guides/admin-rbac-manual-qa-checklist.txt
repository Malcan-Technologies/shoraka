=== General RBAC behavior checklist ===

Run these for each test role/user above (at least for the roles that have relevant permissions for that page).

[ ] Login as **Super Admin** and confirm you can access every admin page/route listed below.
[ ] Login as **View-only Admin**:
  [ ] Confirm sidebar/menu items show only for permissions ending with `.view`.
  [ ] Directly open any admin route that requires a missing `.view` permission (paste URL in browser) and confirm **Access Denied** appears.
  [ ] Confirm read-only pages load successfully when `.view` exists.
  [ ] Confirm any mutation/action buttons are present but **disabled/greyed out** when `.manage` is missing, and the tooltip/helper text says:
    - `You do not have permission to perform this action.`
[ ] For each role, attempt an action by clicking a disabled/greyed-out button (or calling the same action URL/API if applicable) and confirm the backend returns **403 Forbidden**.
[ ] Confirm that when `.view` is missing, page data is not loaded (no spinner stays forever; page should show Access Denied, and network calls should not succeed).
[ ] Confirm frontend hiding + backend enforcement are consistent:
  [ ] If frontend disables/hides an action due to missing permission, backend must also block it with 403.
[ ] Confirm Application comments remain allowed when only `applications.view` is present (no requirement for `applications.manage`).

=== Super Admin lockout protection ===

Setup: Have exactly one active Super Admin (User A) and one other admin with a different role (User B).

Role deletion
  [ ] Open Settings > Roles > Permission Configuration
  [ ] Try to delete the Super Admin role
  [ ] Expected: blocked - backend returns 403, UI hides or disables the delete button

Last Super Admin deactivation
  [ ] Log in as a user with roles.manage
  [ ] Go to Settings > Roles (admin users table)
  [ ] Find User A (the only active Super Admin)
  [ ] Click Deactivate on User A
  [ ] Expected: button is disabled with tooltip "At least one active Super Admin must remain."
  [ ] If you bypass the UI and call the API directly, expect HTTP 400

Last Super Admin role change
  [ ] In the admin users table, find User A (only active Super Admin)
  [ ] Click the role edit icon on User A
  [ ] Expected: edit is blocked with error "At least one active Super Admin must remain. Assign another Super Admin first."
  [ ] If you bypass the UI and call the API directly, expect HTTP 400

Second Super Admin present
  [ ] Assign Super Admin role to User B
  [ ] Now deactivate User A (no longer the last Super Admin)
  [ ] Expected: deactivation succeeds
  [ ] Confirm User B remains active with full access

Pending invitations do not count
  [ ] Have exactly one active Super Admin (User A)
  [ ] Create a pending Super Admin invitation (do not accept it yet)
  [ ] Attempt to deactivate User A
  [ ] Expected: deactivation is still blocked (pending invitation does not count)

Super Admin permission edit protection
  [ ] Open Permission Configuration for the Super Admin role
  [ ] Attempt to uncheck permissions and save
  [ ] Expected: backend returns 403 - Super Admin retains full access regardless
